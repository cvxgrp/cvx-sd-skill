**The strongest validation signal in the whole project — empirical, not
conceptual.** Everything else is *reasoning about* whether the skill is
well-designed; this is evidence it works: the author **already uses this pattern
in their own active research**, unprompted, as **adaptable boilerplate for a lot
of their initial analysis steps.**

**Why it matters:**
- True dogfooding — the author finds the artifact useful for the work the
  artifact is about. Skills that survive contact with their author's real
  workflow are a small, healthier subset.
- WHAT is useful confirms the design is *true*, not just internally consistent:
  it's used as an *adaptable starting substrate you modify* (compose), NOT a
  canned solution (shop). The usage pattern matches the "specify only somewhat /
  compose-don't-shop" philosophy exactly ([[convex-not-brittle-underspecification]],
  [[beer-eno-epigraph]]).
- Answers the earlier adoption-interest hedge ("narrow but deep"): value at the
  deep end is demonstrated, even if interest is niche. Value and reach are
  different axes.

**Implications:**
1. `philosophy.md` can speak from **earned confidence** — frame as "a
   systematization of a working practice," not "a tool we hope is useful."
2. **Best source for `examples/`** is the author's *real* initial-analysis
   boilerplate, sanitized + domain-annotated (the way PV was). Real-practice
   examples read differently from feature-demo examples. Ask the author which
   boilerplate can be shared.
3. The author's own adaptability is the **acceptance test for scaffold
   ergonomics** ([[graceful-degradation-model-capability]]): if the author finds
   it adaptable, the ergonomics are landing.

→ tone of `philosophy.md`; sourcing of `examples/`.
